A Small Note About Simple Software

Published on 2026-05-16

Simple software is often described as boring, limited, or unfinished. I think this is unfair. Good simple software is not software with fewer ideas. It is software where the important ideas are easier to see.

A static blog is a good example. It does not need a database, an administration panel, a JavaScript framework, or a deployment pipeline made of twelve different services. It can be a directory of files, a stylesheet, and a small build script.

The value of boring structure

A boring structure is useful because it stays understandable over time. If the layout of the project can be explained by listing a few directories, the project is easier to debug, move, archive, and rebuild.

For example, this structure is enough for many personal blogs: 

blog/
├── index.html
├── css/
│   └── style.css
├── articles/
│   └── example-article.html
├── md/
│   └── example-article.md
└── templates/
    └── article.html

There is nothing clever here. That is the point. The source article lives in md/, the generated article lives in articles/, and the style lives in css/.

Short lines matter

Text is not more readable just because the screen is wider. Long lines make it harder for the eye to return from the end of one line to the beginning of the next. A comfortable article layout usually keeps the main text column narrow enough that reading feels calm.

A readable page does not try to fill all available space. It leaves room for the reader to breathe.

This is why the stylesheet uses a content width based on characters, such as 65ch, instead of allowing paragraphs to stretch across the full browser window.

A small checklist

For a plain personal blog, I would keep the first version limited to these features:

  • one homepage with a chronological list of articles;
  • one generated HTML file per article;
  • one shared CSS file;
  • one RSS feed, added later;
  • no JavaScript unless there is a real need for it.

That gives you a site that is easy to host almost anywhere. It also means the site can be rebuilt from source files without depending on a runtime application.

Example command

The article can be generated from Markdown with a command like this: 

pandoc md/example-article.md \
  --template=templates/article.html \
  --standalone \
  -o articles/example-article.html

The exact command can change later, but the idea remains the same: Markdown is the source format, HTML is the published format.

Final thought

The best first version of a personal blog is probably not the most feature-rich one. It is the one that makes publishing feel frictionless. If writing an article means opening a Markdown file, running one command, and copying a few static files to a server, that is already a strong foundation.
